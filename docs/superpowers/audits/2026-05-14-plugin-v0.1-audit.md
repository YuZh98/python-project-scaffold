## Verdict
Request changes

## Executive summary
One major behaviour bug: the two sibling skills `release-helper` and `changelog-normalizer` disagree on the canonical subheading order, so they will rewrite each other's output on every release. Two additional majors (PR-ref misattribution in the plugin's own CHANGELOG; spec/SKILL.md drift on that same subheading order) plus a small cluster of minors. Fix the subheading-order constant in `normalize_changelog.py` (and its SKILL.md prose) before tagging — the bug ships green CI today but will manifest the first time anyone runs both skills against the same file.

## Cross-cutting observations
- `surface_detect.py` is intentionally conservative — its docstring (line 21-23) calls out the bias toward over-flagging. The `return_type_narrowed` regex `->\s+\w+\s*:` (line 86) fires on every annotated function definition including test `-> None` methods. The user brief flagged these as false positives; the script's own contract says that's fine and the coordinator filters. No finding, but worth noting: if false-positive volume ever degrades the audit signal, tighten the regex to require a paired removal-line check.
- The two skills `release-helper` and `changelog-normalizer` form a tight contract (release rotates, normalizer validates) but ship without an end-to-end test that runs release.sh then normalize --check on the resulting file. F1 would have been caught by such a test. Primary fix lives in `correctness` (F1); the gap is a test-discipline observation.
- The plugin advertises itself as ready to ship (v0.1.0 manifest, marketplace entry, README) but the scaffold itself has no `pyproject.toml`, so release-helper cannot release the scaffold via the very plugin in this repo. Documented elsewhere (spec §5.2 implicitly via `pyproject.toml` precondition); not a finding, but the plugin's "use me on any Python project" framing should call this out in `plugins/new-project/README.md`.
